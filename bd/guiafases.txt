# GUÍA DE IMPLEMENTACIÓN POR FASES
## Sistema de Gestión - Tienda de Electrodomésticos

Cada fase incluye: objetivo, tablas involucradas, endpoints del backend, pantallas del frontend, reglas de negocio, validaciones, pruebas y el hito que marca la fase como completada.

## FASE 0 — Setup Técnico

**Objetivo:** Tener la base donde escribir todo el código y la conexión a la BD funcionando.

**Tareas:**
1. Elegir y instalar el stack tecnológico (backend + frontend + ORM).
2. Crear repositorio en Git y .gitignore (ignorar `.env`, `node_modules`, `vendor`, `storage/logs`).
3. Crear estructura de carpetas: `backend/`, `frontend/`, `docs/`, `database/`.
4. Configurar `.env` con: `DB_HOST`, `DB_PORT`, `DB_NAME=bd_sistema_electrodomesticos`, `DB_USER`, `DB_PASSWORD`, `JWT_SECRET`, `APP_URL`.
5. Instalar drivers de MySQL/MariaDB y probar conexión.
6. Definir estructura estándar de respuesta API:
   ```json
   { "success": true, "code": 200, "message": "OK", "data": {...} }
   ```
7. Configurar CORS, body parser, manejo global de errores.
8. Configurar logger (winston / monolog).
9. Importar los 3 scripts SQL: `sistema_electrodomesticos.sql`, `permisos_sistema.sql`, `datos_semilla.sql`.

**Hito de completado:** Servidor backend levantando, frontend mostrando una página de prueba, conexión a BD verificada con un SELECT al usuario admin.

## FASE 1 — Autenticación y Seguridad

**Objetivo:** Que solo usuarios autenticados puedan usar el sistema y solo accedan a lo que su rol permite.

**Tablas:** `usuarios`, `sesiones`, `roles`, `permisos`, `rol_permiso`, `usuario_sucursal`, `auditoria`.

**Backend — Endpoints:**
- `POST /api/auth/login` — recibe username + password, valida con bcrypt, genera JWT, registra en `sesiones` y `auditoria` (acción LOGIN).
- `POST /api/auth/logout` — invalida sesión, registra LOGOUT en auditoría.
- `POST /api/auth/cambiar-password` — usuario logueado cambia su propia contraseña.
- `GET /api/auth/me` — devuelve datos del usuario actual, su rol, sucursales asignadas y lista de permisos.
- `POST /api/auth/seleccionar-sucursal` — si tiene acceso a varias.

**Backend — Middlewares:**
- `authMiddleware`: valida JWT, carga `req.user` con id, rol, sucursales.
- `permisoMiddleware('codigo.permiso')`: verifica que el rol del usuario tiene ese permiso.

**Frontend — Pantallas:**
- Login (username + password + botón "Ingresar").
- Cambiar contraseña (forzar en primer ingreso).
- Selector de sucursal (si aplica).
- Manejo global de token expirado (redirigir a login).

**Reglas:**
- Bloquear usuario tras 5 intentos fallidos en 15 minutos.
- Token JWT con expiración de 8 horas.
- Contraseñas con mínimo 6 caracteres, bcrypt cost 10.
- `debe_cambiar_pass=1` redirige obligatoriamente a cambio.

**Pruebas:**
- Login válido → recibe token.
- Login inválido → error.
- Endpoint protegido sin token → 401.
- Endpoint sin permiso → 403.

**Hito de completado:** Pantalla de login funcional, usuario admin puede entrar y al recargar mantiene sesión.

## FASE 9 — Compras

**Objetivo:** Manejar todo el ciclo de compras desde el pre-pedido hasta el pago al proveedor.

**Tablas:** `compras`, `compra_detalle`, `compra_cuotas`, `pagos_compra`, `productos`, `stock`, `kardex`.

**Backend — Endpoints:**
- `GET/POST/PUT /api/compras`
- `POST /api/compras/:id/confirmar-pedido` (PRE_PEDIDO → POR_LLEGAR + genera cuotas si crédito)
- `POST /api/compras/:id/recibir` (recibe mercadería, actualiza stock y kardex)
- `POST /api/compras/:id/anular`
- `GET /api/compras/:id` (detalle completo)
- `POST /api/pagos-compra` (registra pago a proveedor)
- `DELETE /api/pagos-compra/:id` (anula pago)

**Frontend — Pantallas:**
- "Compras": lista con filtro por estado, proveedor, fecha.
- Nueva compra: 3 estados visibles (PRE_PEDIDO / POR_LLEGAR / RECIBIDA), header (proveedor, fecha, depósito destino, moneda, condición), detalle (productos + cantidades + precios), totales.
- Recepción: marcar cantidades recibidas (puede ser parcial).
- "Pagos a Proveedores": registrar pago seleccionando cuota o pago completo, método (EFECTIVO/QR/TRANSFERENCIA), banco, subir comprobante.

**Reglas:**
- Pre-pedido NO afecta stock.
- Por_Llegar genera cuotas si es CREDITO (n cuotas según `dias_credito`).
- Al RECIBIR: por cada producto recibido, INSERT en kardex (tipo COMPRA), UPDATE stock (cantidad += recibida), recalcular costo_promedio.
- Si pago es CONTADO, marcar pagado al recibir; si CREDITO, saldo = total.
- Auditar todos los cambios de estado.
- Subir comprobante (foto/PDF) obligatorio para pagos por transferencia o QR.

**Hito de completado:** Se hace una compra a ROSVANIA, llega, se recibe y el stock sube en el depósito destino. Se paga la compra y baja el saldo del proveedor.

## FASE 10 — Inventario - Operaciones

**Objetivo:** Mover mercadería entre depósitos y hacer ajustes por conteo físico.

**Tablas:** `transferencias`, `transferencia_detalle`, `ajustes_inventario`, `ajuste_inventario_detalle`, `stock`, `kardex`.

**Backend — Endpoints:**
- `GET/POST /api/transferencias`
- `POST /api/transferencias/:id/enviar`
- `POST /api/transferencias/:id/recibir`
- `POST /api/transferencias/:id/anular`
- `GET/POST /api/ajustes-inventario`
- `POST /api/ajustes-inventario/:id/aprobar`
- `POST /api/ajustes-inventario/:id/anular`

**Frontend — Pantallas:**
- "Transferencias": lista con estados (SOLICITADA, EN_TRANSITO, RECIBIDA, PARCIAL, ANULADA). Nueva: selector origen/destino, productos + cantidades.
- Pantalla de recepción de transferencia (almacenero destino confirma).
- "Ajuste de Inventario": elegir depósito, sistema muestra todos sus productos con cantidad sistema vs cantidad física (input), motivo, guardar como borrador → aprobar.

**Reglas:**
- Al solicitar transferencia: stock origen `cantidad_reservada += X`.
- Al enviar: kardex SALIDA + stock origen `cantidad -= X, reservada -= X`.
- Al recibir: kardex ENTRADA destino + stock destino `cantidad += recibida`.
- Si recibe menos de lo enviado: estado PARCIAL.
- Ajuste aprobado: kardex AJUSTE +/- por cada diferencia + UPDATE stock con `cantidad_fisica`.

**Hito de completado:** Se transfieren productos del almacén Gallo18 a Victoria y el stock se mueve correctamente.

**Objetivo:** El motor del negocio — vender al por mayor y al por menor.

**Tablas:** `ventas`, `venta_detalle`, `venta_cuotas`, `pagos_venta`, `devoluciones_venta`, `devolucion_venta_detalle`, `productos`, `stock`, `kardex`, `promociones`, `combos`.

- `POST /api/ventas/:id/anular`
- `POST /api/ventas/:id/pdf` (factura)
- `GET /api/ventas/:id/preview` (vista previa antes de emitir)
- `POST /api/ventas/agregar-producto-rapido` (alta rápida de producto y agrega a la venta en curso)
- `POST /api/devoluciones-venta`
- `POST /api/devoluciones-venta/:id/aprobar`

**Frontend — Pantallas (replicando tu UI):**

**Cabecera de venta:**
- Usuario (logueado), Abrir/Cerrar Caja, Cliente, Factura, Hora, Fecha, Código, Cantidad, Producto, Stock.

**Detalle:**
- Factura, Cliente, Producto, Precio Unit, Impuesto, Total.

**Opciones laterales (productos):**
- Agregar → modal con: Producto, Código, Proveedor, Precio, Precio Final, Costo, Stock, Categoría, Sucursal, Estado, Impuesto. Botones: Guardar / Cancelar / Guardar Imagen.
- Editar.
- Eliminar.
- Limpiar (vaciar carrito).
- Precio (modificar precio del ítem en la venta).
- Agregar Producto que no está en mi BD (alta rápida).

**Opciones abajo:**
- Efectivo (cobro inmediato).
- Vista Previa.
- Agregar Producto.
- Guardar (como borrador o emitida según estado).
- Agregar Cliente.

**Precio a Pagar:** total + IVA + descuentos aplicados + total final.

Selector visible: "Por Mayor 4 Puntos en Red" / "Por menor 4 Puntos en Red" (cambia precio_publico vs precio_mayor).

**Reglas:**
- Si `PERMITIR_VENTA_SIN_STOCK=false`, validar stock disponible en el depósito de la venta.
- Si CREDITO, validar `(cliente.saldo_actual + total) <= cliente.limite_credito`.
- Aplicar promociones vigentes automáticamente.
- Al emitir: por cada ítem, INSERT en kardex (SALIDA), UPDATE stock (cantidad -= vendida), guardar costo_unitario snapshot.
- Si CONTADO + EFECTIVO: marcar PAGADA, registrar `pagos_venta`, sumar a arqueo de caja.
- Si CREDITO: generar `venta_cuotas`, sumar a `cliente.saldo_actual`.
- Recalcular alertas de stock mínimo después de la venta.

**Pruebas:**
- Venta menor al contado en efectivo → stock baja, caja sube.
- Venta mayor a crédito → cuotas se generan, saldo del cliente sube.
- Aplicar combo y verificar precio especial.
- Aplicar promoción de 10% en CONSUL y verificar descuento.
- Intentar vender más de lo que hay → bloquear.

**Hito de completado:** Se realiza una venta completa con cobro y la factura se imprime. **Aquí ya puedes salir a producción.**

- `POST /api/cobros` (registrar cobro de cuota o venta completa)
- `PUT /api/cobros/:id`
- `DELETE /api/cobros/:id` (anular)
- `GET /api/cobros/:id/recibo` (genera PDF de recibo)
- `GET /api/cobros/cuentas-por-cobrar` (clientes con saldo)

**Frontend — Pantallas:**
- "Cobros": tabla paginada con < Anterior / > Siguiente, búsqueda, filtros por:
  - Tipo: Contado / Crédito.
  - Estado: Pendiente / Pagado / Vencido.
  - Fecha, hora.
  - Cliente: Nombre, NIT/CI, Dirección, Ciudad, Celular.
- Botones: Ingresar / Eliminar / Modificar / Grabar / Exportar.
- Método de pago: Efectivo / QR.
- Vista de recibo de pago.

**Reglas:**
- Al registrar cobro: UPDATE cuota (monto_pagado), UPDATE venta (saldo_pendiente), UPDATE cliente (saldo_actual).
- Si cuota se completa: estado PAGADA.
- Si venta queda en cero: estado PAGADA.
- Cobro en efectivo afecta arqueo de caja del turno actual.
- Auditar cualquier anulación de cobro.

**Hito de completado:** El cobrador puede entrar a la pantalla "Cobros", buscar clientes con deuda y registrar pagos.

## FASE 15 — Caja

**Objetivo:** Controlar el dinero físico de cada sucursal por turno.

**Tablas:** `cajas`, `arqueos_caja`, `pagos_venta`, `gastos`.

**Backend — Endpoints:**
- `GET/POST/PUT/DELETE /api/cajas`
- `POST /api/arqueos/abrir` (apertura de turno)
- `POST /api/arqueos/:id/cerrar` (cierre)
- `GET /api/arqueos/actual` (caja abierta del usuario actual)
- `GET /api/arqueos/:id` (detalle con todos los movimientos del turno)

**Frontend — Pantallas:**
- "Apertura de Caja": modal pidiendo monto inicial.
- "Cierre de Caja": muestra monto sistema calculado + input monto físico + diferencia + observación.
- "Historial de Arqueos": tabla con filtros por sucursal/usuario/fecha.

**Reglas:**
- Solo una caja abierta por usuario/caja a la vez.
- Cálculo:
  ```
  monto_cierre_sistema = monto_apertura
                       + SUM(pagos_venta EFECTIVO del turno)
                       - SUM(gastos EFECTIVO del turno)
                       - SUM(pagos_compra EFECTIVO del turno)
  ```
- Diferencia = monto_cierre_real - monto_cierre_sistema (puede ser negativa).
- No permitir nuevas ventas/cobros en efectivo sin caja abierta (configurable).

**Hito de completado:** Se abre caja, se hace una venta en efectivo, se registra un gasto y al cierre el cuadre es correcto.
